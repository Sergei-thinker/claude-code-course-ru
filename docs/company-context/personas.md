# Персоны пользователей TaskFlow

**Глубокое погружение в то, кто использует TaskFlow и почему**

## Обзор

TaskFlow обслуживает три основные персоны пользователей в нашей клиентской базе. Понимание этих персон критично для продуктовых решений - каждая функция должна обслуживать хотя бы одну ключевую потребность персоны.

## Персона 1: Ирина Соколова (Enterprise Admin)

### Быстрые факты

**Имя:** Ирина Соколова
**Возраст:** 38
**Роль:** VP IT Operations
**Компания:** TechCorp (800 сотрудников, Series C)
**Местоположение:** Москва (распределённая команда по России)
**План TaskFlow:** Enterprise

### Предыстория

Ирина возглавляет IT операции для быстрорастущей SaaS компании. Её команда поддерживает 800 сотрудников через 15 инструментов. Она отчитывается CTO и владеет IT бюджетом (150M ₽ ежегодно).

**Карьерный путь:**
- Начинала как специалист IT поддержки
- Выросла до IT менеджера
- Теперь VP IT Operations (6 прямых подчинённых)
- 12 лет в IT, 8 лет в текущей компании

**Образование:**
- Бакалавр информационных систем
- CISSP сертификация (безопасность)

### Цели и мотивация

**Основные цели:**
1. **Сократить разрастание инструментов** - Сейчас используется 25+ инструментов, хочет консолидировать до 10-12
2. **Улучшить безопасность** - Пройти SOC 2 аудит, соответствовать требованиям enterprise клиентов
3. **Контролировать затраты** - Бюджет 150M ₽, нужно показывать ROI на каждый инструмент
4. **Обеспечить команды** - Давать сотрудникам хорошие инструменты без компромисса безопасности

**Метрики успеха:**
- Количество используемых инструментов (вниз)
- Инциденты безопасности (ноль)
- Удовлетворённость сотрудников инструментами (вверх)
- Стоимость инструментов на сотрудника (вниз)

### Болевые точки и фрустрации

**1. Разрастание инструментов вышло из-под контроля**
- Разработка использует Linear
- Маркетинг использует Monday.com
- Продажи использует Salesforce + Asana
- Дизайн использует Notion
- Нет центрального места для просмотра происходящего

**2. Кошмары безопасности**
- Shadow IT (команды покупают инструменты без одобрения)
- Нет видимости, кто имеет доступ к чему
- Бывшие сотрудники всё ещё имеют доступ к инструментам
- Невозможно предоставить audit logs при запросе

**3. Хаос onboarding/offboarding**
- Новый сотрудник? Нужно создать 25 аккаунтов вручную
- Кто-то уходит? Нужно отозвать 25 доступов
- Занимает 2 дня полностью обеспечить кого-то

**4. Нет видимости**
- Не видно, над чем работают команды
- Невозможно отчитаться руководству
- Невозможно спрогнозировать потребности в ресурсах

### Jobs to be Done

**Когда...**
- ...новый сотрудник присоединяется → Обеспечить весь доступ к инструментам в одном месте, автоматически
- ...кто-то уходит → Отозвать весь доступ мгновенно с уверенностью
- ...аудиторы запрашивают логи → Предоставить всеобъемлющий audit trail без паники
- ...оценка новых инструментов → Убедиться, что они соответствуют стандартам безопасности перед покупкой
- ...CFO спрашивает о затратах на инструменты → Показать ROI и метрики использования для каждого инструмента
- ...инцидент безопасности происходит → Определить масштаб доступа и отозвать немедленно

### Функции TaskFlow, которые их волнуют

**Обязательные:**
- ✅ SSO (Single Sign-On) через SAML - не подлежит обсуждению
- ✅ Audit logs - кто что делал, когда, почему
- ✅ Продвинутые права - role-based access control
- ✅ Массовое управление пользователями - добавление/удаление многих пользователей одновременно
- ✅ Admin дашборд - видимость по всей организации

**Желательные:**
- Кастомные поля пользователей (отдел, местоположение, центр затрат)
- Аналитика использования (кто использует, кто нет)
- Экспорт данных (владейте своими данными)
- 99.9% uptime SLA

**Не волнуют:**
- Фантазийные UI анимации
- Социальные функции
- Публичные доски
- Мобильное приложение (роль desktop-first)

### Цитаты

*"Мне нужно одно место, где я могу видеть, кто имеет доступ к чему, и отозвать его мгновенно при необходимости."*

*"Безопасность не обсуждается. Если у вас нет SSO и audit logs, мы даже не можем вас рассматривать."*

*"Я устала быть узким местом для каждой оценки инструментов. Мне нужны инструменты, которые просто работают и соответствуют нашим стандартам."*

*"Когда сотрудник уходит, я теряю сон, задаваясь вопросом, отозвали ли мы весь его доступ. Это не должно быть так сложно."*

## Персона 2: Артём Николаев (IC Engineer)

### Быстрые факты

**Имя:** Артём Николаев
**Возраст:** 29
**Роль:** Senior Software Engineer
**Компания:** GrowthLabs (150 сотрудников, Series B)
**Местоположение:** Санкт-Петербург (полностью удалённо)
**План TaskFlow:** Pro (по всей компании)

### Предыстория

Артём - senior инженер, фокусирующийся на backend системах. Он кодирует 6-8 часов в день, посещает 2-3 встречи в неделю, работает через 4 часовых пояса с распределённой командой.

**Карьерный путь:**
- Выпускник bootcamp (2018)
- Junior → Mid → Senior инженер (5 лет)
- Специализируется на Node.js, PostgreSQL, AWS

**Образование:**
- Coding bootcamp (Hack Reactor)
- Самоучка (до bootcamp)

### Цели и мотивация

**Основные цели:**
1. **Глубокая работа** - Длинное, непрерывное время фокуса для решения сложных проблем
2. **Ясные приоритеты** - Точно знать, над чем работать, без догадок
3. **Минимизировать переключение контекста** - Меньше инструментов, меньше прыжков
4. **Выпускать качественный код** - Гордость мастерством

**Метрики успеха:**
- Функции, выпущенные за sprint
- Количество багов (низкое)
- Время code review (быстрое)
- Время фокуса (6+ часов ежедневно)

### Болевые точки и фрустрации

**1. Путаница с приоритетами**
- Множественные источники истины (Slack, email, standup, инструмент задач)
- Конфликтующие приоритеты от разных stakeholders
- Неясно, что "срочное" реально означает
- Время, потраченное впустую на вопрос "над чем мне работать?"

**2. Недостаточный контекст**
- Название задачи: "Исправить баг логина" (какой баг? где? почему?)
- Нет технических деталей в требованиях
- Продуктовым спецификациям не хватает граничных случаев
- Приходится охотиться за фоновой информацией

**3. Перегрузка встречами**
- Standups могли бы быть асинхронными
- Обновления статуса прерывают фокус
- Встречи запланированы во время пика времени фокуса
- Часовые пояса делают планирование хуже

**4. Медленные, раздутые инструменты**
- Время загрузки страниц > 3 секунд
- Тяжёлые, тормозящие интерфейсы
- Слишком много кликов для простых вещей
- Мобильный веб непригоден для использования

### Jobs to be Done

**Когда...**
- ...начинаю работу → Увидеть задачи с наивысшим приоритетом немедленно, с полным контекстом
- ...заблокирован на чём-то → Сообщить о блокере без планирования встречи
- ...завершаю задачу → Знать, что дальше, без вопросов
- ...ревьюирую PR → Видеть контекст связанной задачи без выхода из GitHub
- ...планирую мощность → Видеть предстоящую работу и оценивать усилия

### Функции TaskFlow, которые их волнуют

**Обязательные:**
- ✅ Горячие клавиши - навигация без мыши (Cmd+K, j/k навигация)
- ✅ Быстрая производительность - загрузки страниц меньше секунды, никаких спиннеров
- ✅ GitHub интеграция - PR связаны с задачами, автообновление статусов
- ✅ Rich markdown - блоки кода, подсветка синтаксиса
- ✅ Контекст на задачах - ясное почему, а не только что

**Желательные:**
- Тёмный режим (ночное кодирование)
- Офлайн режим (работа в самолётах)
- API доступ (автоматизация)
- CLI инструмент

**Не волнуют:**
- Визуальные проектные доски (предпочитают вид списка)
- Графики и диаграммы
- Социальные функции
- Видеозвонки

### Цитаты

*"Просто скажите мне, что строить, дайте контекст, и позвольте мне кодировать. Не заставляйте меня охотиться за информацией или сидеть на встречах."*

*"Если ваш инструмент медленный, я его не буду использовать. Я найду более быструю альтернативу или построю свою."*

*"Мне нужны горячие клавиши для всего. Если мне нужно использовать мышь, я уже раздражён."*

*"Лучшие standups - это те, которые не происходят. Просто дайте мне прочитать обновления асинхронно."*

## Персона 3: Сергей Кузнецов (Team Lead)

### Быстрые факты

**Имя:** Сергей Кузнецов
**Возраст:** 35
**Роль:** Engineering Manager
**Компания:** DataFlow (200 сотрудников, Series B)
**Местоположение:** Новосибирск (распределённая команда через 5 часовых поясов)
**План TaskFlow:** Pro
**Размер команды:** 8 инженеров (3 senior, 5 mid-level)

### Предыстория

Сергей управляет распределённой инженерной командой. Бывший senior инженер, повышен до менеджера 2 года назад. Всё ещё кодирует иногда (~20% времени), но в основном фокусируется на продуктивности команды, найме и доставке.

**Карьерный путь:**
- Инженер (5 лет)
- Tech lead (2 года)
- Engineering manager (2 года)
- Цель: Director of Engineering

**Образование:**
- Бакалавр Computer Science
- MBA (вечерняя программа, в процессе)

### Цели и мотивация

**Основные цели:**
1. **Успех команды** - Команда доставляет вовремя, с качеством, без выгорания
2. **Ясная видимость** - Знать статус без вопросов, выявлять блокеры рано
3. **Сбалансированная нагрузка** - Никто не перегружен, никто не недоиспользован
4. **Карьерный рост** - Развивать членов команды, нанимать A-игроков, вырасти до Director

**Метрики успеха:**
- Sprint velocity (последовательная, предсказуемая)
- Счастье команды (удержание, вовлечённость)
- Частота доставки вовремя
- Качество кода (низкая частота багов)

### Болевые точки и фрустрации

**1. Нет видимости без вопросов**
- Приходится пинговать членов команды: "Каков ваш статус?"
- Standups - это обновления статуса (должны быть асинхронными)
- К моменту всплытия блокеров они критичны
- Невозможно видеть баланс нагрузки, пока не слишком поздно

**2. Постоянное переключение контекста**
- Проверка 5 инструментов для понимания статуса
- GitHub (код), TaskFlow (задачи), Slack (коммуникация), Figma (дизайны), Notion (документы)
- Сложно получить большую картину
- Время, потраченное впустую на агрегацию информации

**3. Отчётность вверх болезненна**
- Руководство спрашивает: "Мы на треке?"
- Занимает 2 часа собрать данные и создать обновление
- К моменту отчёта данные устарели
- Ручная работа, которая должна быть автоматической

**4. Проблемы баланса команды**
- Некоторые инженеры перегружены (работают ночами)
- Другие недоиспользованы (ждут задач)
- Сложно видеть мощность с первого взгляда
- Назначение задач - это догадки

### Jobs to be Done

**Когда...**
- ...планирую sprint → Видеть мощность команды, назначать работу справедливо
- ...кто-то заблокирован → Выявлять блокеры без ежедневного standup
- ...отчитываюсь руководству → Показывать прогресс команды ясно и быстро
- ...1:1 с инженером → Видеть их работу в контексте, предоставлять обратную связь
- ...возникает потребность в найме → Показать руководству, что мы на мощности (нужны headcount)

### Функции TaskFlow, которые их волнуют

**Обязательные:**
- ✅ Дашборд команды - задачи всех с первого взгляда
- ✅ Вид нагрузки - кто перегружен, у кого есть мощность
- ✅ Видимость заблокированных задач - красные флаги, срочное внимание
- ✅ Отчёты по спринтам - velocity, burndown, предсказуемость
- ✅ Сводки комментариев - догнать обсуждения без чтения всего

**Желательные:**
- Предиктивная аналитика ("команда завершит на 2 дня позже при текущем темпе")
- Историческая velocity (сравнивать спринты)
- Отслеживание индивидуальной производительности (для ревью)
- Интеграция с календарём (видеть, кто в отпуске, кто доступен)

**Не волнуют:**
- Детальные детали задач (доверяю команде обрабатывать)
- Фантазийные анимации
- Социальные функции

### Цитаты

*"Мне нужно знать, на треке ли моя команда, без вопросов им индивидуально каждый день. Это не масштабируется."*

*"Когда я в back-to-back встречах весь день, мне нужно догнать прогресс команды за 5 минут, а не за 2 часа."*

*"Я доверяю своей команде, но мне нужна видимость. Если кто-то заблокирован и я не знаю об этом, это на мне."*

*"Отчётность вверх не должна быть ручным процессом. Данные есть - просто покажите их мне в формате, которым я могу поделиться."*

## Как персоны влияют на продуктовые решения

### Пример: Функция тёмного режима

**Перспектива Ирины:**
- Не волнует (роль desktop-first, работает обычные часы)
- Не блокер, но не драйвер

**Перспектива Артёма:**
- **ЛЮБИТ ЭТО** (кодирует поздно ночью, уменьшает напряжение глаз)
- Будет активно адвокатировать за TaskFlow из-за этого
- Дифференциатор vs конкурентов

**Перспектива Сергея:**
- Хорошо иметь (некоторые инженеры работают поздно)
- Не критично для успеха команды
- Сделает команду счастливой

**Решение:** Выпустить тёмный режим
- Высокая ценность для персоны Артёма (удержание + адвокация)
- Низкая стоимость создания
- Позитивный сигнал рынку (современный инструмент)

### Пример: Продвинутые права

**Перспектива Ирины:**
- **ОБЯЗАТЕЛЬНО** (не может купить без этого)
- Блокер для enterprise сделок
- Готова платить премию

**Перспектива Артёма:**
- Не волнует (не его работа)
- Пока это не замедляет инструмент

**Перспектива Сергея:**
- Полезно (контролировать, что команда видит)
- Не критично

**Решение:** Выпустить продвинутые права
- Блокер для enterprise сегмента (высокая выручка)
- Базовая функция
- Требуется для расширения upmarket
