# 3.2.2: Визуализация стратегии и архитектуры

- **Время на выполнение:** 20 минут
- **Необходимые знания:** Визуализация пользователей и продукта (Модуль 3.2.1)

> **Начните этот модуль в Claude Code:** Выполните `/start-3-2-2` для запуска интерактивного обучения.

## Обзор

Модуль 3.2.2 научит создавать визуализацию стратегии и архитектуры для разных аудиторий стейкхолдеров. CTO хочет понять техническую архитектуру. CEO хочет видеть приоритизацию. Board хочет видеть roadmap. Разные аудитории, разные визуализации.

**Ключевой вывод:** Стратегические визуализации переводят сложные идеи в мгновенную ясность. Правильная диаграмма может достичь за секунды того, чего 10-минутное объяснение не может.

## Сценарий

Ваш pitch TaskFlow Mobile прошёл. Теперь у вас последующие встречи:

1. **Встреча с CTO** - "Как это интегрируется с нашими существующими системами?"
2. **Встреча с CEO** - "Что мы строим первым и почему?"
3. **Встреча с Board** - "Каков 3-квартальный roadmap?"

Каждой нужен другой тип визуализации.

## Диаграммы системной архитектуры

### Что они такое

Диаграммы системной архитектуры показывают, как компоненты соединяются - сервисы, базы данных, API, внешние интеграции. Они отвечают "как работает техническая система?"

### Когда использовать

- Технические deep-dives с инженерами
- Презентации CTO/VP инженерам
- Ревью безопасности и соответствия
- Обсуждения планирования интеграции

### Как генерировать

**Базовый промпт:**
```
Создай диаграмму системной архитектуры, показывающую [систему].
Компоненты: [список компонентов]. Показать поток данных между ними.
Чистый технический стиль, подходящий для [аудитории]. 16:9 соотношение сторон.
```

**Пример - TaskFlow Mobile:**
```
Создай диаграмму системной архитектуры для TaskFlow Mobile.

Компоненты:
- TaskFlow Mobile app (новое)
- TaskFlow Backend API (существующее)
- PostgreSQL база данных
- Redis кэш
- Firebase (push уведомления)
- Mixpanel (аналитика)

Показать стрелки потока данных между компонентами.
Чистый технический стиль, подходящий для ревью CTO. 16:9 соотношение сторон.
```

## Матрицы приоритизации

### Что они такое

2x2 матрицы размещают items на двух измерениях - обычно Impact vs Effort, или Value vs Complexity. Они визуализируют решения приоритизации.

### Когда использовать

- Обсуждения приоритизации функций
- Executive стратегические встречи
- Сессии планирования
- Выравнивание стейкхолдеров

### Классическая 2x2

Самая распространённая: **Impact vs Effort**

```
Высокое влияние │  Крупные    │  Быстрые
                │  проекты    │  победы
────────────────┼─────────────┼──────────
Низкое влияние  │  Избегать   │  Заполнители
                │             │
                └─────────────┴──────────
                 Высокое усилие Низкое усилие
```

| Квадрант | Значение | Действие |
|----------|---------|--------|
| **Быстрые победы** | Высокое влияние, низкое усилие | Делать первым |
| **Крупные проекты** | Высокое влияние, высокое усилие | Планировать тщательно |
| **Заполнители** | Низкое влияние, низкое усилие | Делать, когда время позволяет |
| **Избегать** | Низкое влияние, высокое усилие | Не делать |

### Как генерировать

**Базовый промпт:**
```
Создай 2x2 матрицу приоритизации.
X-ось: [измерение 1] (Низко → Высоко)
Y-ось: [измерение 2] (Низко → Высоко)

Метки квадрантов: [список квадрантов]

Размести эти items:
- [Item 1]: [квадрант]
- [Item 2]: [квадрант]
...

Чистый корпоративный стиль для презентации руководству. 16:9 или 1:1.
```

## Product Roadmaps

### Что они такое

Product roadmaps показывают запланированную работу во времени - обычно по кварталам или месяцам. Они отвечают "каков план и когда?"

### Когда использовать

- Презентации Board
- Ежегодные ревью планирования
- Выравнивание команды
- Обновления внешних стейкхолдеров

### Структуры Roadmap

| Структура | Лучше всего для |
|-----------|----------|
| **Timeline** | Чёткий хронологический вид |
| **Swimlane** | Несколько параллельных треков |
| **Now/Next/Later** | Гибкий, без дат |
| **Milestone** | Фокус на ключевых deliverables |

### Как генерировать

**Базовый промпт:**
```
Создай product roadmap для [продукта].

Timeline: [временной период]

[Период 1]: [Тема]
- [Item 1]
- [Item 2]

[Период 2]: [Тема]
- [Item 1]
- [Item 2]

...

Профессиональный стиль для [аудитории]. 16:9 соотношение сторон.
```

**Пример - TaskFlow Mobile:**
```
Создай product roadmap для TaskFlow Mobile на 3 квартала.

Q1 - MVP запуск:
- Основное управление задачами
- Push уведомления
- Offline просмотр

Q2 - Командные функции:
- Real-time синхронизация
- Интеграция team чата
- Общие списки задач

Q3 - Enterprise:
- Админ контроли
- SSO интеграция
- Продвинутая аналитика

Профессиональный стиль, подходящий для презентации board.
Показать чёткую прогрессию от MVP к enterprise-ready.
16:9 соотношение сторон.
```

## Советы для конкретных стейкхолдеров

### Для CTO/Инженеров

- **Фокус на:** Точки интеграции, поток данных, технические зависимости
- **Включить:** Существующие системы, API, базы данных
- **Стиль:** Технический, но читаемый; не overwhelming
- **Уровень:** Архитектурный, не детали реализации

### Для CEO/Руководства

- **Фокус на:** Стратегические решения, обоснование приоритизации
- **Включить:** Компромиссы, что вы НЕ делаете
- **Стиль:** Чистый, минимальный, высокоуровневый
- **Уровень:** Бизнес влияние, не технические детали

### Для Board

- **Фокус на:** Timeline, milestones, стратегическое направление
- **Включить:** Фазы, ключевые deliverables, бизнес результаты
- **Стиль:** Профессиональный, полированный, уверенный
- **Уровень:** Квартальный, не недельный

## Использование вашей библиотеки стилей

Стратегические визуализации невероятно выигрывают от постоянного стилирования.

**Сохранить ваши любимые стили:**
- "Executive Architecture" - чистые технические диаграммы
- "Corporate Matrix" - профессиональные 2x2
- "Board Roadmap" - полированные timeline визуализации

**Затем переиспользовать:**
```
Сгенерируй системную архитектуру, используя стиль #34
Сгенерируй матрицу приоритизации, используя стиль #5
Сгенерируй roadmap, используя стиль #88
```

Постоянство через материалы стейкхолдеров выглядит профессионально.

## Лучшие практики

### Делать:
- **Соответствовать стиль аудитории** - технический для инженеров, чистый для руководства
- **Рассказывать чёткую историю** - каждая визуализация должна иметь одно основное сообщение
- **Использовать постоянное стилирование** - использовать вашу библиотеку стилей
- **Держать просто** - макс 5-10 элементов на визуализацию
- **Итерировать** - первые черновики редко финальные

### Не делать:
- **Не усложнять** - 5-10 элементов на визуализацию макс
- **Не смешивать метафоры** - выбрать один визуальный подход и придерживаться его
- **Не пропускать метки** - всё должно быть чётко идентифицировано
- **Не использовать жаргон неосторожно** - соответствовать словарю аудитории
