- **Мобильное приложение TaskFlow** - новая вещь, которую мы строим
- **Backend API TaskFlow** - существующий backend
- **Слой базы данных** - PostgreSQL для данных, Redis для кэширования
- **Сторонние интеграции** - push-уведомления, аналитика
- **Поток данных** - как информация движется между компонентами

Это тот вид диаграммы, который отвечает на "как все это соединяется?"

## Часть 2: Стратегический фреймворк (встреча с CEO)

Дальше: встреча с CEO.

CEO хочет увидеть, как вы приоритизируете функции - что строится первым и почему. Матрица 2x2 идеальна для этого: **Impact против Effort**.

Давайте составим карту ключевых функций для TaskFlow Mobile:

- **Quick Wins** (Высокое воздействие, Низкие усилия): Push-уведомления для обновлений задач, офлайн-просмотр задач
- **Major Projects** (Высокое воздействие, Высокие усилия): Синхронизация в реальном времени между устройствами, интеграция командного чата
- **Fill-ins** (Низкое воздействие, Низкие усилия): Кастомные темы, поддержка виджетов
- **Avoid** (Низкое воздействие, Высокие усилия): Поддержка голосовых команд, AR-визуализация задач

СТОП: Попросите меня создать матрицу приоритизации 2x2 для функций TaskFlow Mobile.

ПОЛЬЗОВАТЕЛЬ: Создай матрицу приоритизации / Сгенерируй матрицу 2x2

ДЕЙСТВИЕ: Сгенерируйте матрицу приоритизации 2x2
- Ось X: Effort (Низкие → Высокие)
- Ось Y: Impact (Низкое → Высокое)
- Четыре квадранта с метками: Quick Wins, Major Projects, Fill-ins, Avoid
- Разместите функции, упомянутые выше, в соответствующих квадрантах
- Чистый корпоративный стиль, подходящий для презентации руководству (стиль #5 или похожий)
- Соотношение 1:1 или 16:9
- Сохраните в папку outputs с описательным именем

ДЕЙСТВИЕ: Предоставьте путь к выходному изображению, чтобы пользователь мог просмотреть его.

Проверьте папку `outputs/` для вашей матрицы приоритизации.

СТОП: Видите, как матрица делает логику приоритизации визуальной и четкой?

ПОЛЬЗОВАТЕЛЬ: Ответ

ДЕЙСТВИЕ: Если пользователь хочет изменения, итерируйте на матрице. Если доволен, продолжайте.

Теперь CEO может видеть с первого взгляда, что вы строите первым и почему.

## Важные заметки для Claude

- **Пути к файлам:** Всегда предоставляйте абсолютные пути, когда говорите пользователям, где найти файлы.
- **Генерация изображений:** Используйте модуль `image_gen.py` через Python. Начните новую сессию в начале каждой основной генерации, чтобы держать сессии чистыми.
- **Библиотека стилей:** Если пользователь ссылается на номер стиля, посмотрите промпт в `style-library.html` (в папке модуля 3.1.4).
- **Итерация:** Если любое изображение не получается хорошо, предложите регенерировать или улучшить его. Используйте сессию для итерации.
- **Именование вывода:** Используйте описательные имена вроде `taskflow_system_architecture.png`, `feature_prioritization_matrix.png`, `product_roadmap_q1_q3.png`.
- **Техническая точность:** Для диаграммы архитектуры убедитесь, что компоненты и соединения логически имеют смысл для мобильного приложения backend.
- **Открытие изображений:** Если пользователь испытывает трудности с поиском изображения, предложите открыть его за него, используя `open [path]` (Mac) или `start [path]` (Windows).
